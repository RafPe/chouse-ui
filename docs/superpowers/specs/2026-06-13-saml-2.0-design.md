# SAML 2.0 SSO — Design

**Status:** Approved (brainstorming) · **Date:** 2026-06-13
**Builds on:** the existing OIDC/OAuth2 SSO subsystem (`packages/server/src/rbac/sso/`).

## Goal

Add SAML 2.0 as a third SSO provider type, supporting **both SP-initiated and
IdP-initiated** login, reusing the existing provisioning pipeline. v1 includes
consuming IdP metadata; it excludes SP metadata publishing, encrypted assertions,
SP-signed requests, and Single Logout.

## Scope

**In v1:**
- SP-initiated and IdP-initiated login.
- Mandatory signed-assertion verification against a configured IdP cert.
- Consume IdP metadata (paste URL or XML → auto-fill fields).
- SAML attribute → identity mapping and group-attribute → role mapping.
- JIT provisioning + auto-link-by-email, reusing `provisionSsoUser` unchanged.
- Admin wizard branch + env/YAML config parity + the three `sso:*` permissions.

**Out of scope (v1):** SP metadata endpoint · encrypted assertions ·
SP-signed AuthnRequests (no SP private key) · Single Logout (SLO) ·
HTTP-Artifact binding · multi-instance shared stores (single-instance only).

## Library

`@node-saml/node-saml` (the maintained core behind passport-saml). Chosen for a
small native-dep surface (`xml-crypto` + `@xmldom/xmldom` + `xpath`, on Node
`crypto` which Bun implements) → best Bun-compatibility odds. XML-DSig verification
is delegated entirely to the library (never hand-rolled), which carries the
signature-wrapping defenses.

**Task 0 (gating spike):** confirm under Bun that node-saml (a) verifies a
known-good signed assertion and (b) parses IdP metadata. If it fails, fall back to
a small Node sidecar for verification or re-evaluate `samlify`. The full plan is
gated on this spike.

## Architecture & integration seam

SAML is `type: 'saml'` in the existing Zod discriminated union, stored in the same
`rbac_sso_providers` table, surfaced in the same Admin → SSO section and login
button list (the global master switch + per-provider `enabled` still gate it).

New, isolated modules under `packages/server/src/rbac/sso/saml/`:
- `saml/client.ts` — wraps `@node-saml/node-saml`: build the AuthnRequest
  (SP-initiated) and validate a `SAMLResponse` (both flows) → produce `SsoIdentity`.
- `saml/metadata.ts` — parse pasted IdP metadata (URL or XML) →
  `{ idpEntityId, idpSsoUrl, idpCertificate }`.

**The seam is `SsoIdentity`** (`{ provider, subject, email, emailVerified,
username, displayName, claims }`). SAML adds only an identity-acquisition path;
`provisionSsoUser` and `syncMappedRoles` are reused verbatim. Mapping:
- `subject` = SAML **NameID** (override via `claim_mapping` `subject:<attr>`).
- `email` / `username` / `displayName` = configured SAML **attributes**.
- `emailVerified` = **`true`** — the assertion is signed by the trusted IdP, so its
  asserted email is authoritative; this enables auto-link-by-email. (Explicit
  decision; documented in code.)
- `claims` = the full SAML attribute set (drives role mapping + the debug log).

## Data model (migration `1.31.0`, both dialects)

New **nullable** columns on `rbac_sso_providers`:
- `saml_idp_entity_id` — IdP entityID (the assertion Issuer).
- `saml_idp_sso_url` — IdP SSO endpoint (HTTP-Redirect binding).
- `saml_idp_certificate` — IdP X.509 signing cert (PEM/base64).
- `saml_sp_entity_id` — our SP entityID / audience (defaults to `baseUrl`).
- `saml_nameid_format` — requested NameID format.
- `saml_allow_idp_initiated` — boolean; gate the unsolicited flow per provider.

**Reused columns:** `display_name`, `enabled`, `claim_mapping`
(attr → subject/email/username/displayName), `role_mapping_claim` (group attribute
name), `role_mapping` (group → role).

**NOT NULL relaxation:** `client_id`, `client_secret_encrypted`, and `scopes` are
currently `NOT NULL` but meaningless for SAML. The migration relaxes all three to
nullable; per-type requiredness is enforced at the Zod/route layer (already
type-aware). Relaxing `NOT NULL` is safe for existing OIDC/OAuth2 rows; SAML rows
leave them null.

`APP_VERSION` → `1.31.0`. `VERSION_CHECKS["1.31.0"]` asserts the new columns exist
and the three columns are nullable, on SQLite and PostgreSQL.

## Endpoints & flows

**`/start` (reused, polymorphic).** `GET /rbac/auth/sso/:provider/start` branches on
type. For SAML it builds an AuthnRequest and 302-redirects (HTTP-Redirect binding)
to `saml_idp_sso_url`, with **RelayState** carrying the existing signed state
(provider id + safe redirect target). The login button is unchanged.

**ACS (new).** `POST /rbac/auth/sso/saml/acs` — a **single** endpoint for all SAML
providers. The IdP form-POSTs `SAMLResponse` (+ optional `RelayState`). The provider
is resolved by matching the assertion's **Issuer (IdP entityID)** to a configured
SAML provider. This is the one ACS URL the admin registers at the IdP:
`${baseUrl}/auth/sso/saml/acs`.
- SP-initiated: validate `InResponseTo` against the state we issued (via RelayState).
- IdP-initiated: no `InResponseTo`; permitted only if `saml_allow_idp_initiated`.

**Token handoff (one-time code).** OIDC's `/callback` is a SPA `fetch` returning JSON
tokens; the SAML ACS is a browser form-POST and cannot return JSON to JS. So the ACS:
1. validates, mints tokens, stores them server-side keyed by a single-use code
   (TTL ~60s), and 302-redirects to the SPA route `/login/sso-complete?code=…`;
2. the SPA calls `POST /rbac/auth/sso/saml/exchange` with the code → receives
   `{ user, tokens, redirect }` → stores tokens exactly as today (Zustand) →
   navigates to the redirect.

This keeps the SPA's bearer-token model and the OIDC flow unchanged. New pieces: the
ACS handler, an in-memory handoff-code store (TTL, single-use), the `exchange`
endpoint, and the `/login/sso-complete` SPA route.

## Security model

- **Mandatory signature verification** against `saml_idp_certificate`; reject
  unsigned/invalid. XML-DSig delegated to node-saml (covers signature-wrapping).
- **Audience restriction**: assertion `Audience` must equal `saml_sp_entity_id`.
- **Replay protection**: cache assertion IDs until their `NotOnOrAfter`; reject
  reuse. Essential for IdP-initiated (no `InResponseTo`). In-memory TTL cache.
- **Conditions** (`NotBefore`/`NotOnOrAfter`) enforced with a small
  `acceptedClockSkewMs`.
- **InResponseTo**: SP-initiated must match an AuthnRequest we issued; IdP-initiated
  must have none and be explicitly enabled.
- **RelayState redirect** restricted to same-app relative targets (reuse
  `safeRedirect`).
- Global master switch + per-provider `enabled` gate everything (reused).
- **No SP private key in v1** (no signed requests, no encryption): trust rests
  solely on the configured IdP cert.

## Admin UI (wizard SAML branch)

- Type select adds **SAML**.
- **Step 1 — Identity:** displayName, enabled; a read-only "SP details to register at
  your IdP" block (ACS URL, SP entityID) and an editable SP entityID (defaults to
  `baseUrl`). No client id/secret.
- **Step 2 — SAML config:** a "Paste IdP metadata (URL or XML)" box that auto-fills
  entityID / SSO URL / cert, or manual fields (IdP entityID, SSO URL, signing cert
  textarea, NameID format select, allow-IdP-initiated switch); then attribute mapping
  (`claim_mapping`) + role mapping (reusing existing editors).
- **Step 3 — Review & validate:** static validation only (cert/metadata parse,
  required fields present); a note that full verification happens on first sign-in.
  Save does not require a live test for SAML.
- Provider rows, enable/disable, delete, and `sso:view/edit/delete` are reused.

## Config / YAML parity

`AUTH_SSO_PROVIDERS_<ID>_TYPE=saml` plus `_SAML_IDP_ENTITY_ID`, `_SAML_IDP_SSO_URL`,
`_SAML_IDP_CERTIFICATE`, `_SAML_SP_ENTITY_ID`, `_SAML_NAMEID_FORMAT`,
`_SAML_ALLOW_IDP_INITIATED`, plus shared `_DISPLAY_NAME` / `_CLAIM_MAPPING` /
`_ROLE_MAPPING_CLAIM` / `_ROLE_MAPPING`. New suffixes added to the loader's
`FIELD_SUFFIXES` + `FIELD_TO_KEY`. The multi-line cert is supported as base64 or
`\n`-escaped. Env-sourced providers remain read-only in admin (reused).

## Testing

- **Task 0 spike** (gating): node-saml verifies a known-good signed assertion +
  parses metadata under Bun.
- **Unit:** `saml/client` (AuthnRequest contents; validate signed fixture →
  `SsoIdentity`; reject tampered / unsigned / wrong-audience / expired / **replayed**;
  IdP-initiated allow vs block), `saml/metadata` (parse → fields), `config` Zod
  (env + DB; SAML in `buildSsoConfig`), `store` (SAML columns round-trip), routes
  (`/start` SAML redirect, ACS → handoff code, `exchange`, Issuer-resolution,
  IdP-initiated gating).
- **Migration tests (mandatory):** `VERSION_CHECKS["1.31.0"]` on SQLite + PostgreSQL.
- **Fixtures:** a self-signed IdP keypair to deterministically sign test assertions.
- **Frontend:** `rbacSsoAdminApi` SAML type + light wizard-branch coverage.

## Error handling

Reuse `describeSsoError` and the honest disabled-vs-unknown patterns. SAML validation
failures (bad signature, audience, expiry, replay, unknown Issuer, IdP-initiated
disabled) map to clear, non-leaky `/login?ssoError=…` redirects and verbose
server-side `requestLogger.warn` lines; the new full-claims debug log applies once an
assertion validates.

## Known limitations (v1)

- Single-instance only: handoff-code store and replay cache are in-memory. Multi-
  instance deployments need shared storage (Redis/DB) — flagged, not built.
- No SLO: signing out is local-only (does not propagate to the IdP).
